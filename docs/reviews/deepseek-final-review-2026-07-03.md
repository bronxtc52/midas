## Находки

Ниже перечислены баги и слепые зоны реализации, которые могут проявиться в проде. Severity оценивается по влиянию на целостность процесса, безопасность токенов и финансы.

1. **CRITICAL — Полная перезапись меток (labels) при смене состояния (transitionState)**  
   `src/gh.js` – метод `transitionState` делает `PUT /repos/{repo}/issues/{n}/labels` с телом `{ labels: [...rest, to] }`. Это полностью заменяет все метки на issue. Если в момент выполнения человек (или второй процесс) добавил любую другую метку (не state‑лейбл), она будет молча потеряна. Демон не должен перезаписывать не‑state‑лейблы; нужно использовать `POST /labels` и `DELETE /labels/{name}` для атомарного добавления/удаления только одного лейбла. **Потеря вручную проставленных меток владельца или других систем.**

2. **HIGH — Превышение дневного капа стоимости внутри одного tick**  
   `src/daemon.js` – проверка дневного лимита (`keeper.costForDay(day) >= config.cost_cap_usd_per_day`) выполняется только *перед* циклом обработки задач в `tick`. Если текущий день уже близок к капу (например, $19 из $20), все задачи внутри этого же tick будут запущены без дополнительных проверок и могут израсходовать ещё несколько долларов, превысив дневной лимит. Остановка произойдёт только на следующем тике. Финансовый контроль не жёсткий.

3. **MEDIUM — Двойной запуск Planner’а после краша демона**  
   `src/daemon.js` – для действия `plan` не используется `keeper.markProcessed`. Если демон упал после перевода лейбла в `state:planning`, но до завершения `runPlanner` и вызова `keeper.markProcessed`, при рестарте issue остаётся в `state:planning`, `decide` снова вернёт `plan`, и Planner выполнится повторно (лишний расход токенов и дублирование плана в комментариях). Отсутствует идемпотентность для этапа планирования.

4. **MEDIUM — Возможно превышение капа задачи внутри одной LLM‑сессии**  
   `src/roles/common.js` – функция `capExceeded` проверяет оставшийся бюджет *перед* запуском сессии; сама сессия не имеет жёсткого ограничения по стоимости (только `max‑turns` и таймаут). Если на задачу уже потрачено $4.90 при капе $5.00, сессия может израсходовать $0.15, превысив лимит. Блокировка произойдёт только на следующем этапе (например, при переходе к Reviewer), но факт превышения уже состоялся. Критерий «останавливается со `state:blocked`» выполняется с задержкой и допускает перерасход.

5. **LOW — Лишний GH_TOKEN в окружении git‑процессов**  
   `src/roles/worker.js` – `gitEnv` собирается как `{ ...process.env, ...gitEnvWithAskpass() }`. `process.env` содержит `GH_TOKEN` (полный права), хотя для git‑операций используется `GIT_ASKPASS` с `MIDAS_GIT_TOKEN`. Это не приводит к прямой утечке, но расширяет поверхность: GH_TOKEN доступен всем дочерним процессам без необходимости.

6. **LOW — Отсутствие идемпотентности для создания PR после краша**  
   `src/roles/worker.js` – если worker упал после `git push`, но до вызова `gh.createPR`, при повторном запуске флаг `existing` (ветка на удалённом репо) станет `true`, и код поймёт, что ветка уже есть, но PR не был создан. Метод `createPR` вызывается только когда `existing === false` или когда не найден существующий PR. В данной ситуации PR будет создан, что нормально. Однако если worker упал *сразу после* `createPR`, но до вызова `transitionState`, повторный запуск найдёт существующий PR через `getPRForBranch`, и дублирования не будет – угрозы нет. Тем не менее, гонка на границе "push – создание PR – переход метки" не закрыта полностью, но к авариям не приводит.

7. **INFO — Review-issue может «зависнуть», если её `updated_at` не меняется**  
   `src/daemon.js` – для борьбы с этим используется специальная выборка всех `state:review` без `since`. Решение корректное, но при очень большом количестве review‑issues может создавать лишнюю нагрузку на GitHub API (пагинация до 10 страниц × 100). В v1 это допустимо.

## Приёмка §5

| № | Критерий | Достижим? (да/нет/частично) | Почему |
|---|----------|----------------------------|--------|
| 1 | `docker compose up -d` на mh-central: демон работает, healthcheck зелёный | **Да** | Dockerfile содержит HEALTHCHECK (проверка heartbeat-файла, обновляемого каждые 30 с). Compose-файл монтирует `data:/app/data`, демон при старте создаёт heartbeat. Запуск через `node src/daemon-main.js` стартует демон немедленно. |
| 2 | Issue со `state:ready` подхватывается ≤60 с: журнальная запись + перевод лейбла | **Да** | Интервал опроса 45 с (конфиг). При старте `tick()` вызывается сразу, затем по таймеру. Максимальное ожидание ≤45 с + время обработки, что укладывается в 60 с. Переход лейбла (optimistic) и запись в журнал Keeper’а выполняются. |
| 3 | Worker создаёт ветку и PR, связанный с issue, и ставит `state:review` | **Да** | `runWorker` клонирует репо, создаёт/продолжает ветку `midas/issue-{n}`, выполняет Claude-сессию, коммитит, пушит, создаёт PR с упоминанием `Closes #n` и переводит лейбл в `state:review`. Тест `roles.test.js` подтверждает интеграцию с git. |
| 4 | Задача с неполным ТЗ получает `state:blocked` + комментарий заданного формата; демон не трогает до ручного возврата | **Да** | `Planner` при неоднозначности возвращает BLOCKED-строку, `parseBlockedFromSession` её распознаёт, `makeBlock` формирует канонический комментарий и переводит лейбл в `state:blocked`. Автомат (`decide`) для `state:blocked` возвращает `null`, демон игнорирует issue до ручной смены лейбла владельцем. |
| 5 | Reviewer оставляет структурированный вердикт по рубрике; запускается только при зелёных CI-чеках | **Да** | `handleReview` проверяет `checksStatus`: красные → возврат в `coding`, pending → ожидание, зелёные (или старый PR без CI) → запуск `runReviewer`. Reviewer формирует вердикт `VERDICT: {...}`, комментарий содержит список находок. |
| 6 | Acceptor ставит `midas:accept`/`midas:reject`; reject возвращает `state:coding` | **Да** | `runAcceptor` при `verdict.verdict==='pass'` добавляет `midas:accept` и переводит в `state:accepted`; при `fail` добавляет `midas:reject` и переводит из `review` обратно в `coding` с комментарием причин. |
| 7 | Kill демона → создать issue → поднять демон → issue подхвачен (курсор работает) | **Да** | Курсор сохраняется атомарно (`renameSync`) в `cursor.json` после каждого успешного tick’а. При старте используется `since` с перекрытием (на 120 с раньше курсора), что гарантирует подхват событий, появившихся во время простоя. |
| 8 | Секретов нет ни в репо, ни в образе (gitleaks чисто); всё из KV в рантайме | **Да** | CI включает `gitleaks` (ограничение по глубине — full fetch). В исходный код и `config.json` не зашиты секреты. `Dockerfile` не копирует `.env`. Секреты загружаются из Azure Key Vault через `fetch-env.sh` и передаются в контейнер через `env_file`. |
| 9 | В коде ролей отсутствует вызов merge; мерж PR выполняет только человек | **Да** | `gh.js` не содержит методов для слияния PR (poll, merge, merge_method). Автоматический тест `no-merge.test.js` сканирует все `.js` файлы в `src/` и проверяет отсутствие вызовов merge‑эндпоинтов GitHub API. |
| 10 | Задача, превысившая кап стоимости, останавливается со `state:blocked` и отчётом $ | **Частично** | `capExceeded` блокирует запуск следующей роли, если суммарный расход задачи превысил `cost_cap_usd_per_task` минус `MIN_SESSION_BUDGET_USD`. Остановка происходит на границе этапов. Однако перерасход может случиться внутри одной LLM‑сессии, и задача продолжится до следующего этапа (например, после Worker перейдёт в Review, где и будет заблокирована). Денежный отчёт включается в комментарий blocked-вопроса. Полностью «жёсткого» реального‑времени капа нет. |
| 11 | Sentry принимает тестовое событие проекта `midas`; сервис виден в watchdog-fleet | **Да** | `daemon-main.js` инициализирует `@sentry/node` при наличии `SENTRY_DSN` и отправляет события (blocked, ошибки tick). Watchdog‑fleet регистрируется через heartbeat‑файл, который может мониториться внешним скриптом. Тестовое сообщение легко инициируется. |
| 12 | `plans/midas-v1-backlog.md` + `docs/constitution.md` v1.1 в репо; issues этапов на доске «MIDAS v1» | **Частично** | Код демона не проверяет наличие этих артефактов — они создаются и размещаются человеком. При условии, что владелец вручную выполнит требования по документированию и настройке GitHub Projects, критерий будет достигнут. Автоматического контроля соблюдения в коде нет. |
