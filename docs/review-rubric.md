# Review-рубрика — как ревьюить дифф на уровне Fable golden reviews

Дистиллят Fable-спринта t2-1 (43 эталона: 36 обычных + 7 мин, `golden/`). Потребители:
Reviewer-роль (этап 4 `/armanda`), MIDAS, `kb-agent-standards`. Few-shot к промпту —
`golden/_fewshot-pack.md`.

> ✅ **Веса и топ-3 утверждены владельцем 2026-07-03** (стоп-кран t2-2 пройден). Топ-3
> (вес 3): безопасность/границы доступа · деньги-данные-миграции · регресс-радиус старой
> эвристики. Множитель домена: деньги/authz/платежи/миграции ×1.5.

## Как пользоваться

1. Определи **домен** диффа (см. множитель строгости ниже) — он важнее размера диффа.
2. Пройди 7 измерений сверху вниз; для каждого задай чек-вопросы. Топ-3 (вес 3) —
   обязательны всегда; средние (вес 2) — когда дифф их касается.
3. Ранжируй найденное по `вес измерения × множитель домена`, а не по порядку в файле.
4. Вердикт по критериям ниже. Не пиши стилевой шум (анти-паттерны в конце).

**Множитель строгости по домену** (калибровка из вердиктов t2-1: 29 approve / 7
request-changes, все RC — в деньгах/авторизации/межпроектных границах):
`деньги · авторизация/приватность · платежи · миграции данных` = **строгий** (×1.5);
`ops/мониторинг · реестры · docs` = **базовый** (×1.0). Один и тот же класс бага в
платежах = request-changes, в ops-скрипте = замечание.

## Измерения (вес в скобках)

### 1. Безопасность и границы доступа — вес 3 (ТОП)
Меняет ли дифф, **кто и что может**? Границу авторизации/приватности снимают под
нейтральным тегом чаще, чем кажется.
- Меняется authz/authn-граница? Тогда это **продуктовое решение владельца** — есть
  согласование и тест на новую границу? (`gr-fc-fb00da6a4d55`: RBAC-гейт approve снят как
  «feature»).
- Секрет/токен/константа идентичности — в коде вместо стора? (`gr-miniapp-19369f6c0e75`:
  BOT_TOKEN в bot.py).
- Тест-фикстура, меняющая состояние через HTTP-endpoint без оплаты/роли → этот endpoint
  достижим и в проде? (`gr-binar-02e78e51eb35`: activate-package без оплаты).
- Инструкция/док, исполняемая агентом буквально, не ведёт к межпроектной краже
  (чужой токен «у соседа»)? (`gr-miniapp-11d6f85ee1f1`).
- App-only/широкий scope (Files.Read.All, весь тенант) за одной env-переменной?
  (`gr-president-732aef35db83`).

### 2. Данные, деньги и миграции — вес 3 (ТОП)
- Денежная семантика: та же валюта/единица? (`gr-miniapp-00e394a0d2cd`/fix: Value нац.
  валюта vs ValueUsd; недоплата/переплата; `'error' ≠ 'pending'`).
- Миграция с `CREATE UNIQUE`/constraint — что будет на **уже грязных** прод-данных? Дубли
  схлопнуты до индекса? (`gr-binar-bde38fb0b4c0`).
- Идемпотентность: клиент может занять предсказуемый системный ключ (`order:{id}`)?
  Двойная оплата / оплата отменённого заказа? Потерянный платёж тихо no-op vs лог+Sentry?
- Радиус разрушения ручной SQL по FK (`cascadeOnDelete` на ledger).

### 3. Регресс и радиус старой эвристики — вес 3 (ТОП)
- Рефактор изменил, **что теперь ловит** старое условие/эвристика? Что стало с
  последствиями его срабатывания? (`gr-mhaibot-349921a4d7f8`: small-talk regex глотает
  вопросы; окно-100 → MAX_PAGES-5000).
- **Семантическая подмена под сохранённым алиасом** — имя поля прежнее, смысл другой; кто
  из потребителей читает старую семантику и обновлён ли он тем же диффом?
  (`gr-miniapp-ed0587fa9653`: Raw_PV, CountryCode Code3→Code, категория по Id vs Number).

### 4. Контракты и парные половины — вес 2
- У изменённой половины контракта есть вторая — обновлена и кем гарантируется? (cron `7-22`
  vs окно `[7,22)`; health-URL vs жизненный цикл FQDN — `gr-sw-144352be5dae`,
  `gr-sw-b6f6c874488c`).
- Расширилось множество допустимых входов — умеет ли парный слой новый вход или он молча
  схлопывается? (`gr-fc-a1b269667ce1`: одиночный конец периода игнорируется).
- Имя поля декларирует свойство, которое потребитель не может проверить (фронт печатает
  `amount_usd`, доверяя бэку) — ловится только сквозным ревью пары.
- Добавлен метод в интерфейс → все реализации вне диффа не сломаются фатально?

### 5. Направление отказа и устойчивость — вес 2
- Когда предпосылка защиты не выполнена — ломается в **безопасную** сторону? (fail-open:
  пустой GH-секрет → тихий деплой без Sentry — `gr-marineweb-e980b18574d9`; fail-closed:
  isAdmin при пустом env). Направление выбрано осознанно?
- Таймер/интервал, дёргающий внешний ресурс (БД/сокет), переживает его закрытие в другой
  фазе? (`gr-mhaibot-933f47208f0b`: `setInterval(_cleanup)` на закрытой БД; guard
  `if(!handle)` не спасает, если handle не-null, а соединение мёртво).
- Observability доставляет событие на пути, где процесс может умереть?
  (`gr-president-7850c290818a`: captureException без `Sentry.close()`).
- Бурст запросов (пагинация без паузы) сам не словит rate-limit? MAX_PAGES/срез не режет
  тихо на самом большом (важном) случае?

### 6. Тесты и закрепление проверок — вес 2
- Что из проверенного **руками** закреплено так, что регресс поймается без этих рук?
  («tester 8 scenarios green» / «прогнал на postgres:16» без ассерта в диффе —
  `gr-sw-144352be5dae`, `gr-binar-bde38fb0b4c0`).
- Новая граница авторизации/поведения покрыта тестом? (RBAC на одном `router.use` без
  негативного теста — `gr-mhaibot-38d00baeefb9`).
- Инфра-действие **выполнимо из репо** или только описано в commit message (out-of-band
  роли → при пересоздании ресурса дыра вернётся молча — `gr-sw-94e665d892c7`)?

### 7. Форматирование вывода, инъекции, санитайзер — вес 2
- LLM-текст наружу → `html.escape` + markdown→формат канала во **всех** точках вывода?
- Размеченный текст (HTML/Telegram) не режется срезом `[:N]` посреди тега?
  (`gr-sw-8d0357415def`).
- Интерполяция в DOM/HTML экранирована (stored-XSS)? (`gr-mhaibot-38d00baeefb9`: targetId
  без escape). SQL — параметризован, не f-string по недоверенному входу?
- LLM с полным KB в system-prompt — вход пользователя не переопределяет инструкции
  (prompt injection)? (`gr-binar-02e78e51eb35`).

## Сквозные модификаторы (не отдельное измерение)
- **«Временное» ослабление без механизма снятия** (continue-on-error, override без плана
  апгрейда) — всегда спрашивать «что заставит это убрать?» (`gr-binar-66a07055e73d`,
  `gr-marineweb-bfe78a14149b`).
- **Фикс экземпляра вместо класса** — «где живёт следующий экземпляр и что мешает ему
  появиться?» (`gr-marineweb-5daed077f5b2`).
- **Docs-диффы ревьюятся как код** — центральный факт верифицируем; правь живые доки, но
  историю (backups/handoff) только дополняй tombstone, не переписывай
  (`gr-mhaibot-057b360c4676`, `gr-president-141076e9ec4d`).

## Позитив-сигналы (отмечать, если несут сигнал — симметрично запрету шума)
- Верификация факта в commit message (метод + дата: `id sadmin`→no such user —
  `gr-president-95809d29ebc0`).
- Комментарий объясняет **почему**, а не что (`gr-binar-bde38fb0b4c0`: данные перед
  индексом, куда деваются деньги).
- Направление отказа выбрано осознанно и проговорено.

## Критерии вердикта
- **Approve** — нет находок веса 3 без смягчения; веса 2 терпят follow-up; денежная/
  authz-семантика сохранена. Уверенность: высокая, если ключевой инвариант виден в диффе;
  средняя, если требует контекста вне диффа.
- **Request-changes** — любая находка веса 3 в строгом домене без смягчения; снятие
  границы доступа без согласования/теста; необратимая потеря данных/денег; XSS/инъекция/
  утечка секрета. Всегда с механизмом → последствием → рекомендацией, не «мне не нравится».
- **Уверенность обязательна** в вердикте (низкая/средняя/высокая) + 2–3 строки обоснования.

## Анти-паттерны ревью (запрещены)
- **Стилевой шум**: «переименуй переменную», формат, порядок импортов — если не влияет на
  риск. ТЗ t2-1 это прямо запрещает.
- **Ложная уверенность**: утверждать баг, не показав механизм срабатывания; «выглядит
  небезопасно» без сценария вход→последствие.
- **Симметрия шума наоборот**: хвалить каждую строку. «Сделано хорошо» — только если несёт
  сигнал (закреплённый инвариант, осознанное направление отказа).
- **Строгость по размеру диффа**, а не по домену: 4К-строк фронт-переписка в registry
  мягче, чем 3-строчное снятие RBAC-гейта в платежах.
